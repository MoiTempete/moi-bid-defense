# 投标讲标答辩幻灯片生成

## 这个 Skill 做什么

根据用户提供的投标技术响应文件（md 或 docx），自动完成：

1. **解析响应文件** — 提取章节结构、★ 条款对应、量化指标清单、架构关键词
2. **黄金线索提取** — 按"痛点回应 + 差异化亮点 + 量化冲击力"三维评分模型，从数万字响应中提取最具讲标价值的内容
3. **生成幻灯片大纲** — 按叙事节奏编排，标注时间分配、讲标备注、预判 Q&A
4. **逐页编写幻灯片** — 每页 = 标题 + 3-5 条要点 + 数据高亮 + 讲标备注 + 预判 Q&A，用户逐页确认
5. **输出幻灯片** — 先输出 Markdown 大纲供确认，再按用户选择渲染为 guizang-ppt HTML 或标准 PPTX

## 输入要求

用户需要提供：

- **投标技术响应文件路径**（必需）：.md 或 .docx 格式的已完成的技术响应文件
- **讲标时长**（可选，默认 20 分钟）：决定幻灯片页数预算
- **评委构成信息**（可选）：技术评委为主 / 商务领导为主 / 混合型，影响侧重点
- **差异化方向**（可选）：用户认定的核心竞争优势，用于指导提取重点

## 关键概念

### 三种讲标叙事策略

根据响应文件特征和评委构成，选择合适的叙事策略：

**策略 1 — 技术纵深型**（适合技术评委为主）：
从痛点切入 → 架构总览 → 核心技术模块逐层展开 → 性能指标举证 → 实施保障。以技术深度和专业性取胜。

**策略 2 — 价值引领型**（适合商务领导为主）：
从客户需求理解切入 → 我们带来的核心价值 → 关键场景演示 → 服务承诺 → 案例佐证。以共鸣感和可信度取胜。

**策略 3 — 攻防兼备型**（适合混合评委，默认推荐）：
需求理解 → 架构一览 → 选 3-4 个最亮眼的模块深讲 → 一期继承策略 → 实施团队与计划 → 服务与交付承诺。每个技术亮点都标注对应的商务价值。

### 幻灯片元素模型

每页幻灯片由五个元素构成：

| 元素 | 说明 | 示例 |
|------|------|------|
| **标题** | 一个完整的判断句或核心主张 | "AI 三大能力中枢——全私有化部署，甲方可自主训练" |
| **要点** | 3-5 条，每条不超过两行，用数字和动词驱动 | "12 类视频 AI 算法：明火 ≥98%、烟雾 ≥95%、安全帽 ≥97%" |
| **数据高亮** | 1 个最核心的数字，放大展示 | **200%** 设计容量 |
| **讲标备注** | 给演讲者的提示：这句话怎么讲、停顿点、眼神交流时机 | "此处停顿 2 秒，目光扫过评委席，然后说——" |
| **预判 Q&A** | 1-2 个评委可能追问的问题及应答要点 | "Q: GPU 成本多少？A: 训练用一张 A100-80G 即可起步，约 8-10 万" |

### 黄金线索三维评分模型

这是本技能的核心算法，用于从响应文件中提取最具讲标价值的内容。评分维度：

**维度 1 — 痛点回应（Pain Point）**：这段话是否直击客户的核心关切？
- 引用一期不足并给出针对性解决方案：+3 分
- 点出项目的核心挑战/难点：+2 分
- 针对各分公司差异化需求：+2 分
- 涉及化工安全特有风险（H₂S、爆炸、泄漏等）：+2 分
- 涉及政府监管合规要求：+1 分
- 涉及地理分散/系统异构等现实困难：+1 分

**维度 2 — 差异化亮点（Differentiation）**：这段话是"人无我有"还是"人有我优"？
- 独有技术/自主研发/创新方案：+4 分
- 硬承诺（费用全包/免费/终身/知识产权归甲方）：+3 分
- 训练平台/源代码/模型权重完整交付：+3 分
- 私有化部署/数据不出内网：+2 分
- 架构前瞻性（200% 容量/线性扩展/零中断升级）：+2 分
- 低代码/零代码/拖拽式/可视化配置：+2 分
- 标准领先（GB 30871 对齐/LEC+LS 双引擎）：+2 分
- 持续进化（季度迭代/增量训练/自动优化）：+2 分

**维度 3 — 量化冲击力（Quantitative Impact）**：有没有"一句话就能记住"的数字？
- 段落包含 4 个以上量化指标：+2 分
- 段落包含 2-3 个量化指标：+1 分

**扣分项**：
- 纯模板套话且无差异化标签：-2 分
- 仅"满足/符合/遵循"无实质内容：-2 分

### 幻灯片页数预算

| 讲标时长 | 建议页数 | 节奏 |
|---------|:---:|------|
| 15 分钟 | 12-15 页 | 封面 0.5' → 叙事 12' → Q&A 2.5' |
| 20 分钟 | 15-20 页 | 封面 0.5' → 叙事 16' → Q&A 3.5' |
| 30 分钟 | 20-28 页 | 封面 0.5' → 叙事 22' → 演示 5' → Q&A 2.5' |

## 工作流

### Step 1 · 解析响应文件

**目标**：从技术响应文件中提取结构化信息。

**操作**：

1. 如果输入是 .docx，先用 `python3 ~/.claude/skills/moi-bid-defense/scripts/parse_response.py <响应文件.docx>` 解析
2. 如果输入是 .md，直接读取文件内容
3. 从解析结果中获取：
   - 章节层级树
   - ★ 条款引用清单
   - 量化指标表（百分比、时间、规模、速率四类）
   - 架构关键词分布
   - 总体字数统计

4. 展示解析摘要给用户确认：
   ```markdown
   ## 📊 响应文件解析结果
   | 指标 | 数值 |
   |------|------|
   | 总字数 | XX,XXX 字 |
   | 章节数 | XX 个一级章节 |
   | ★ 条款引用 | XX 处 |
   | 量化指标 | XX 个（精度/时间/规模/速率） |
   | 可提取高价值线索 | 预估 XX-XX 条 |
   ```

### Step 4 · 逐页编写幻灯片

**目标**：按大纲顺序，逐页编写幻灯片内容。

#### 编写原则

1. **标题即主张**：每页标题是一个完整的判断句，听众只读标题也能把握要点
2. **要点驱动行动**：每条要点用动词开头或含核心数字，不超过两行
3. **一页一个核心数字**：每页有一个放大的数据高亮
4. **讲标备注是灵魂**：标注停顿点、转折词、眼神交流时机、可能的追问
5. **预判 Q&A 要诚实**：不知道的就说"这个细节我们可以在答疑环节提供书面材料"

#### 每页结构

```markdown
### 第 X 页 · <标题>

⏱ <时间分配>

- <要点 1>
- <要点 2>
- <要点 3>
- <要点 4>（可选）
- <要点 5>（可选）

> 🔢 **数据高亮**：<1 个核心数字>

🎤 **讲标备注**：<给演讲者的提示>

⚡ **预判 Q&A**：
- Q: <评委可能的问题>
- A: <应答要点>
```

#### 每页编写流程

1. 回顾该页对应的黄金线索原文
2. 将原文转化为口述要点（书面语 → 口语化）
3. 提取核心数字作为数据高亮
4. 编写讲标备注
5. 预判 Q&A
6. 展示给用户
7. 等待用户反馈（确认/修改/重写）
8. 处理完后进入下一页

### Step 5 · 输出幻灯片

**目标**：将所有确认的幻灯片内容生成为可播放的演示文件。这是本 skill 的标准交付步骤。

#### 5a. 输出完整 Markdown 大纲

将 Step 4 中已确认的全部幻灯片汇总为一个完整的 Markdown 文件（`讲标幻灯片-<项目名>.md`），供用户最终审核。

#### 5b. 生成投喂版 Markdown（关键中间步骤）

在生成 HTML 幻灯片之前，**必须**对 Markdown 进行编辑处理，将幻灯片画面内容与讲标人备注分离：

1. 每页保留幻灯片展示内容：标题、要点、数据高亮
2. 将讲标备注（🎤）集中移至文件末尾的「讲标人备注」区域
3. Q&A 预判（⚡）放入备注区域，不在幻灯片画面上显示
4. 为每页标注推荐的布局类型（Bullet / Data Hero / 对比 / 时间线 / 架构图等）
5. 保存为 `讲标幻灯片-<项目名>-投喂版.md`

**投喂版格式示例**：

```markdown
## 第 X 页 · 标题（Hero / Bullet / Data Hero 布局）
**幻灯片展示内容**：
- 要点 1
- 要点 2
- **数据大字报**：核心数字

## 讲标人备注（不进入幻灯片画面）
### 第 X 页备注
语速提示、停顿点、眼神交流...
```

#### 5c. 建议生成 HTML 幻灯片（推荐 guizang-ppt）

**如果用户同意使用 guizang-ppt**：

生成过程中需注意：
- 幻灯片画面内容与讲标备注严格分离（备注不渲染到画面）
- 每页的"数据大字报"使用 Data Hero / KPI Hero 布局放大
- 深色背景页慎用——如果视觉效果不好，统一改为白底
- 确保所有文字在投屏场景下可读（正文 ≥18px，描述 ≥16px，标签 ≥14px）
- 生成后执行自动化视觉检查（Step 5d），修复内容稀疏、字号过小、颜色不可读等问题

**如果用户指定其他 PPT 工具**：

按用户指定的工具（如标准 PPTX、Keynote、Google Slides 等）生成对应格式。对于 PPTX，可生成结构化 JSON → `python-pptx` 直接生成。

#### 5d. 自动化视觉检查与修复（必须执行）

在 HTML 生成完成后，**必须**运行以下自动化截图 + 审查 + 修复循环，不再依赖用户手动截图反馈：

##### 5d-1. 自动截图

使用 Playwright + 系统 Chrome 对所有幻灯片逐页截图：

```bash
python3 ~/.claude/skills/moi-bid-defense/scripts/screenshot_slides.py <ppt/index.html路径> <截图输出目录> <页数>
```

- 自动打开 HTML → 逐页截取 1920×1080 PNG → 保存至 `ppt/screenshots/`
- 依赖 `playwright`（`pip3 install playwright`），不需要额外安装 Chromium（使用系统已安装的 Chrome）

##### 5d-2. 读图诊断

截图完成后，使用 Read 工具逐页读取截图，按以下检查清单诊断：

| 检查项 | 判定标准 | 修复方法 |
|--------|---------|---------|
| **字号可读性** | 正文 < 16px、描述 < 14px → 不通过 | 全局 +2px，字重 +100 |
| **标题/正文层级** | 卡片标题 ≤ 正文字号 → 层级反转 | 标题 > 正文至少 2px 且为粗体 |
| **内容密度** | 单页空白区域 > 40% → 太空 | 换更紧凑的 Sxx 版式或扩充描述 |
| **信息过载** | 单行技术栈名称 > 8 个组件 → 不可读 | 精简至 3-5 个核心组件 + "等" |
| **暗底文字** | dark 主题页正文 `rgba(...,<0.9)` → 太暗 | 提至 `.92` 以上 |
| **排版失衡** | 内容堆在左侧 < 50% 宽度 → 布局错误 | 换全宽版式（如 S16/S20 替代 S02 纵向时间线） |
| **KPI 间距** | KPI 区与上方内容 < 1.5vh → 太挤 | padding-top 增至 2vh |
| **文字墙** | 卡片内连续 > 6 行长段落无粗体关键词 → 不可扫读 | 关键数字/术语加 `<b>` 或分段 |
| **条目缺失** | 列表项数与投喂版不一致 → 遗漏 | 检查 grid 行数，补充缺失条目 |
| **深色背景页效果** | 如果视觉效果不好 | 统一改为白底 |

##### 5d-3. 修复与验证

## 关键规则

1. **风格必须先确认**：Step 2 必须完成风格配置，用户确认后才能进入 Step 3
2. **大纲必须等用户确认**：Step 3 生成大纲后不可跳过
3. **逐页确认**：每页写完后展示给用户，不要一次性写完
4. **黄金线索驱动**：幻灯片内容必须以提取脚本的评分排序结果为素材基础，不凭空编造
5. **标题即主张**：每页标题必须是判断句，不能是名词短语
6. **口语化**：书面语转口语——"我方郑重承诺"→"我们承诺"，"本项目"→"这个项目"
7. **一页一个核心数字**：每页只放大一个最核心的量化指标
8. **讲标备注必写**：这是区别于普通 PPT 的核心差异化
9. **Q&A 预判要诚实**：不确定的就说需要书面补充，不强行编造
10. **先 Markdown 后渲染**：Step 5 先输出完整 Markdown 供审核，用户确认后再渲染为最终格式
11. **主动建议 HTML 生成**：Markdown 审核通过后，必须主动询问用户是否生成 HTML 网页幻灯片（推荐 guizang-ppt），不要等用户自己提
12. **guizang-ppt 自动安装**：如果用户同意使用 guizang-ppt 但未安装，展示安装命令并征得同意后自动执行安装，然后调用生成
13. **幻灯片与备注严格分离**：生成 HTML 前必须编辑投喂版 Markdown——画面内容进幻灯片，讲标备注和 Q&A 预判放入文件末尾备注区，不渲染到画面
14. **自动化视觉检查**：HTML 生成后必须执行 Step 5d 的自动截图+诊断+修复循环。使用 `scripts/screenshot_slides.py` 逐页截图，用 Read 工具读图诊断，发现问题直接编辑 HTML 修复后重新截图验证。不允许依赖用户手动截图反馈
15. **支持用户指定工具**：用户可以不选 guizang-ppt，指定其他 PPT 工具（PPTX / Keynote / Google Slides / 任意 skill）
16. **HTML→PPTX 导出**：如需可编辑 PPTX，使用 python-pptx 基于投喂版 Markdown 直接生成，所有文字独立文本框可编辑
