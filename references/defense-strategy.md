# 讲标策略指南

## 叙事策略选择

### 策略 1 — 技术纵深型

**适用场景**：评委以技术专家为主（CTO、技术总监、架构师），关注技术方案的先进性、可行性和细节。

**叙事节奏**：
```
痛点切入(2') → 架构总览(2') → 核心技术模块A(2') → 核心技术模块B(2')
→ 核心技术模块C(2') → 性能指标举证(1.5') → 实施保障(1.5') → Q&A预判(1') → 收尾(0.5')
```

**每页特点**：
- 标题偏向技术判断句："物联网平台采用协议网关集群架构，8 种协议独立伸缩"
- 要点可以适当增加技术细节（框架名、协议名、算法名）
- 数据高亮偏向性能指标（QPS、延迟、可用率）
- Q&A 预判偏向技术深度问题

**风险**：商务领导可能听不懂、失去兴趣。需在收尾页补充商务价值总结。

---

### 策略 2 — 价值引领型

**适用场景**：评委以商务领导为主（分管副总、采购总监），关注投资回报、风险可控、省心省力。

**叙事节奏**：
```
需求共鸣(2.5') → 核心价值主张(2') → 关键场景演示A(2.5') → 关键场景演示B(2.5')
→ 为什么选我们(2') → 服务与保障(1.5') → 案例佐证(1.5') → 收尾(1')
```

**每页特点**：
- 标题偏向商务判断句："让每家分公司都拥有量身定制的安全管理能力"
- 要点减少技术术语，增加"省了什么""避免什么""多快能上线"
- 数据高亮偏向 ROI 相关（省钱%多少、提速几倍、减少多少人力）
- Q&A 预判偏向商务问题（价格、合同、责任）

**风险**：技术评委可能觉得"没干货"。需在关键场景页补充技术亮点卡片。

---

### 策略 3 — 攻防兼备型（默认推荐）

**适用场景**：评委混合型，或不确定评委构成。特点是以一条清晰的主线贯穿，技术深度和商务价值交替呈现。

**叙事节奏**：
```
封面(0.5') → 我们对需求的理解(2') → 核心挑战与应对策略(1.5') → 总体技术架构(2')
→ 亮点深讲A：物联网数据底座(2') → 亮点深讲B：AI智能化(2') → 亮点深讲C：安全生产业务闭环(1.5')
→ 一期继承与平滑升级(1') → 实施计划与团队(1.5') → 服务承诺与交付保障(1')
→ Q&A预判(1') → 收尾(0.5')
```

**每页特点**：
- 技术页附带商务价值标注："这个设计意味着甲方未来新增分公司时，不需要重新采购"
- 商务页附带技术佐证卡片：方框内用 9pt 小字放数据证据
- 讲标备注区分"对技术评委说"和"对商务领导说"两个版本

## 时间分配黄金法则

| 讲标阶段 | 占比 | 20分钟讲标 |
|---------|:---:|:---:|
| 定调（封面+需求理解） | 15% | 3 分钟 |
| 架构（总览+高可用） | 15% | 3 分钟 |
| 亮点深讲（选3-4个） | 35% | 7 分钟 |
| 信任建立（实施+团队+服务） | 20% | 4 分钟 |
| 收尾（总结+Q&A预判） | 15% | 3 分钟 |

⚠️ **不要在架构部分花太多时间**。架构页的目的是让评委知道"你们想清楚了"，不是让他们完全理解架构。3 分钟内必须翻页。

## 评委心理与应对

### 技术评委在想什么

1. "这个方案能落地吗？" → 用实施计划和真实案例回应
2. "有没有技术短板？" → 主动承认（云边协同的延迟控制是难点，但给出了 100ms 的具体指标）
3. "和一期怎么衔接？" → 必须有专门的一期继承页
4. "团队靠谱吗？" → 驻场人数、备份机制、资质证明

### 商务领导在想什么

1. "花了钱能省什么？" → 每个技术亮点都标注对应的管理价值
2. "会不会烂尾？" → 分阶段交付、验收标准、违约金条款
3. "后期费用有多少？" → 明确"免费范围"和"收费范围"
4. "出事谁负责？" → 故障响应时效、兜底承诺、知识产权归属

## 讲标备注编写指南

讲标备注是 `moi-bid-defense` 区别于普通 PPT 生成工具的核心差异。

### 好的讲标备注

```
🎤 讲标备注：
- 前两句话目光扫过评委席，语速放慢
- 念完"99.99%"后停顿 1 秒，让数字沉下去
- 说到"甲方可以自主训练"时提高音量，这是杀手锏
- 如果看到技术评委开始记笔记，放慢这一页的速度
```

### 讲标备注应包含

| 要素 | 示例 |
|------|------|
| 语速提示 | "这一页语速可以快一点，信息量大但都是加分项" |
| 停顿点 | "说完'200%设计容量'停顿 2 秒" |
| 音量提示 | "甲方自主训练——这句话提高音量" |
| 眼神/肢体 | "说到差异化需求时，看向评委席最资深的那位" |
| 翻页时机 | "如果评委开始讨论，不要翻页，让他们讨论" |
| 备用素材 | "如果时间充裕，可以补充一个硫化工H₂S监测的具体例子" |

## Q&A 预判策略

预判 Q&A 不是猜题，而是从响应文件中识别"可能引起质疑"的点，提前准备应答。

### 识别高风险 Q&A 的信号

1. **技术选型可能被质疑**："为什么选 Spring Cloud 而不是 Go/Node.js？"
2. **性能指标过于精确**："99.99% 可用性怎么测算的？"
3. **交付承诺过于慷慨**："源代码都给了，你们靠什么盈利？"
4. **定制比例可能被追问**："35% 定制具体是哪些？有没有超过的风险？"

### 应答策略

- **技术选型质疑** → 不辩护，给理由（生态、人才、可维护性）
- **性能指标质疑** → 给测试方法和验证计划，不空谈
- **商业模式质疑** → 诚实说明（标准产品 + 定制服务 + 年度维保）
- **定制比例质疑** → 列举已经明确的定制项，说明控制机制

### Q&A 模板

```
⚡ 预判 Q&A：
Q: <评委可能的问题>
A: <简短回应（2-3句）+ 如果需要，我们可以在答疑环节提供书面材料>
```

## Swiss 版式实战规则（2026-07 广汇项目沉淀）

以下规则来自真实 23 页讲标 deck 的反复修改，每条背后都有踩过的坑。

### 版式→内容映射

| 讲标内容类型 | 推荐 S 版式 | 不推荐 | 原因 |
|-------------|-----------|--------|------|
| 公司简介/资质展示 | S18 Why Now | — | 三/四列各配一个核心数字，信息密度高且视觉不散 |
| 产品/平台全景（6-8 项） | S17 System Diagram 改造为网格 | S05 五列 | 五列中文必然截断，改用 4×2 grid 每卡 2-3 行描述 |
| 业绩清单（4-6 项有金额） | S20 Stacked Ledger | S06 KPI Tower | 账单式纵向天然适合金额对比，dark 主题凸显数字 |
| 需求理解/差异化分析 | S13 Three Forces | S18 | 左 accent 块做定调 + 右卡片做展开，攻防兼备 |
| 架构总览 | grid-12 自由布局 | S05 三列/五列 | 架构文字密度高，12 列网格给每层足够的描述空间 |
| 物联网/视频AI 规格 | S21 Tech Spec | S19 | 规格页需要标题+KPI+竖线装饰的数据密度 |
| 风险管控/安全模块 | S16 Multi-card Brief | S04 Six Cells | Brief 卡轻量、适合 6 项安全功能快速罗列 |
| 智能巡检/操作卡（双模块） | S08 Duo Compare | S19 四列等宽 | 双栏对比天然契合"巡检 vs 操作卡"的叙事结构 |
| 驾驶舱/数字孪生 | S15 Matrix + Hero Stat | — | 矩阵承载 9 大专题，底部巨数收束 |
| 一期继承/灰度迁移 | S02 Vertical Timeline | S11 Horizontal | 纵向时间轴每节点 2-3 行描述，7 节点仍可全显示 |
| 团队配置 | S06 KPI Tower | S19 | 塔高对比表达经验年限差异，视觉冲击力强 |
| 技术偏离/合规承诺 | S04 Six Cells 改造 | S12 | 32 项 ★ 按类别归入 3×4 grid，比 Manifesto 信息密度高 |
| 实施计划（7 里程碑） | S02 Vertical Timeline | S11 Horizontal | 横向 7 节点文字严重挤压，纵向每节点可承载完整交付物描述 |
| 运维 SLA | 2×2 grid + 3 列底部 | S19 四列等宽 | 四列等宽导致标签列过窄、描述溢出；2×2 给前两列更多空间 |
| 培训体系 | S14 Loop Form | — | 左侧四步纵向均匀分布 + 右侧卡片，justify-content:space-between |
| 核心承诺（7 项） | 4×2 grid + 1 通栏 accent | S12 Manifesto | 承诺点多，矩阵比宣言更清晰 |
| 架构图（大尺寸 PNG） | 左图右文 6:4 分栏 | S22 Image Hero | `object-fit:contain` + 上下布局会让图"顶天"不美观 |
| Q&A 预判 | S04 Six Cells | — | 6 格刚好覆盖 6 个预判问题 |

### 主题节奏规则

- **每 2-3 页切换一次主题**（不超过连续 3 页同主题）
- dark 适合：架构、业绩、团队、承诺——严肃/权威/数据
- light 适合：需求理解、功能介绍、运维、培训——亲和/展开/叙事
- accent（IKB 满屏）只用于封面，split（IKB 半屏）只用于封底
- 新增页面后必须检查相邻页的主题连续性

### 投屏字号底线

| 元素 | 最小 | 推荐 | 备注 |
|------|:---:|:---:|------|
| 页面主标题 | — | `min(3.2vw, 6vh)` 以上 | weight 200 ExtraLight |
| 卡片标题 `.t-cat` | 14px | **16px** / 600 | 投屏时 14px 偏小 |
| 正文 `.t-body` | 16px | **18px** / 400 | |
| 描述 `.t-body-sm` | 15px | **16px** / 400 | |
| 标签 `.t-meta` | **14px** / 500 | — | 绝对底线，不能用 12px |
| KPI 塔内文字 `.sub` | 16px | **18px** / 1.3vw | 塔内描述常被忽略 |

### 常见雷区

1. **dark 主题 + card-fill → 白底白字**  
   `.slide.dark .t-body-sm` 设置 `color:rgba(255,255,255,.78)` 会覆盖 card-fill 内的文字颜色。  
   **修复**：给 card-fill 内的 `t-cat`/`t-body-sm` 显式加 `color:var(--text-primary)`。

2. **五列 stack-row 中文截断**  
   5 列每列约 18vw，中文 5-6 字即换行。改用 `grid-12` + `span-N` 给核心层更多空间。

3. **S19 四列等宽标签溢出**  
   "— 01 / P1 紧急故障" 这种长标签在 1fr 列里截断。改用 2×2 grid 给标签列更多空间。

4. **纵向时间轴 7 节点溢出**  
   默认 `padding:var(--sp-7)`(32px) 每个节点，7 个 224px。显式设置 `padding:1.6vh 0`，yr 列拆两行（M1 + 第X月）。

5. **卡片高度不统一**  
   KPI Tower 所有塔必须同高度（都用 h-4），bar-tower 内 sub 字号统一 `min(24px,3.6vw)`。

6. **插入新页后未更新全局页码**  
   插入页面后必须用脚本批量替换所有 `XX / NN` 页码。

### 应 upstream 到 guizang-ppt 的模板改进

以下问题根源在模板 CSS，建议提交 PR 到 guizang-ppt：

- `.slide.dark .t-body-sm` 级联覆盖了 card-fill 内的文字颜色，应加 `:not(.card-fill *)` 选择器或降低 specificity
- 投屏场景 `t-cat` 默认 14px 偏小，建议新增 `--t-cat-presentation:16px` 变量或在 SKILL.md 中提示
- S05 `.stack-row` 硬编码 `repeat(3,1fr)`，五列需求时需手动 override，建议增加 `data-cols="5"` 支持
